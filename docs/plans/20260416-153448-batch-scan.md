## Problem Statement

三套交易体系（Elder 技术交易、CANSLIM 成长股、Value Investing）目前都缺乏从全 A 股批量筛选标的的能力。

现有的 `signal-scanner` skill 假设候选池已经存在（20-30 只），需要用户手动维护候选标的列表。这不可扩展。

目标：每套体系都能从全 A 股（约 5000 只）批量筛选出值得深度分析的候选清单，输出给对应体系的 skill 层做进一步 AI 判断。

---

## 核心设计原则（来自 blog post: clean-skill-architecture）

**结果确定的步骤用脚本，需要判断的步骤用 AI。**

- 计算 EMA/MACD/随机指标、过滤 EPS 增速 > 25%、筛选 PE 历史分位 → Python 脚本（确定性）
- 判断"这个信号值不值得深入分析"、分析护城河质量 → AI skill（模糊判断）

**明确的交接协议：** scan 命令输出 JSON 文件，skill 层从文件读数据，不关心数据怎么来。

**分层架构：**
```
业务层（各自的筛选逻辑）：scan-elder / scan-canslim / scan-value
通用层（共享数据获取）：全市场快照 + 本地历史数据 + 指标计算
工具层：AKShare / BaoStock / DuckDB
```

---

## 约束

- Python 层新增 CLI 命令，不改现有命令
- 三套体系的 scan 命令独立，不共享筛选逻辑（但共享底层数据获取工具函数）
- 输出格式统一（JSON），skill 层通过约定的文件路径读取
- 前瞻偏差防护：scan 只使用截至昨日收盘的数据，不使用当日实时行情做筛选决策
- scan-elder 全 A 股扫描预期耗时 < 5 分钟（本地计算，不逐只调接口）
- scan-canslim / scan-value 耗时取决于预拉取数据的完整度，不做强制保证

---

## 前提

1. 三套体系的筛选逻辑完全不同，各自独立实现 — 已确认
2. 批量筛选分两阶段：Python 机械过滤（快）→ AI 深度判断（慢）— 已确认
3. 三套 scan 命令共用数据获取层，业务逻辑各自独立 — 已确认
4. 全 A 股日线数据用本地 DuckDB，不逐只实时调接口 — 已确认
5. 输出 JSON 文件作为 skill 层的输入协议 — 已确认

## 方案对比

### 方案 A：仅 Elder 体系（最小可行）
只做 `scan-elder`，快速验证数据层和指标计算。
- 努力：人工 ~2天，CC ~30分钟。完整度：6/10
- 缺点：另外两套体系仍然没有批量筛选

### 方案 B：三套独立 scan 命令（选定）
三个独立子命令，共享数据层，业务逻辑独立。
- 努力：人工 ~1周，CC ~1小时。完整度：9/10
- 优点：架构分层清晰，符合博客里的设计原则

### 方案 C：统一配置化框架
YAML 配置驱动，可扩展到任意体系。
- 努力：人工 ~2周，CC ~2小时。完整度：10/10
- 缺点：过早抽象，现在只有三套体系，配置化的灵活性暂时用不上

## 推荐方案：方案 B 详细设计

### CLI 命令接口

```bash
# Elder 技术交易体系：技术指标过滤
python -m trading_os scan-elder \
  --date 2024-03-15 \        # 默认昨日
  --top 30 \                 # 输出前 N 只（按信号强度排序）
  --min-turnover 1e7 \       # 最低日均成交额（CNY），排除流动性差的股票
  --output artifacts/scan/elder-20240315.json

# CANSLIM 成长股体系：基本面指标过滤
python -m trading_os scan-canslim \
  --date 2024-03-15 \
  --top 30 \
  --output artifacts/scan/canslim-20240315.json

# Value Investing 体系：估值指标过滤
python -m trading_os scan-value \
  --date 2024-03-15 \
  --top 30 \
  --output artifacts/scan/value-20240315.json
```

### 各体系的筛选逻辑

#### scan-elder：技术指标过滤

**数据需求：** 本地 DuckDB 中的日线历史数据（已有）。**周线数据在扫描时从日线 resample 生成**，不单独存储。

```python
# 周线从日线 resample
weekly = daily_bars.resample('W-FRI').agg({
    'open': 'first', 'high': 'max', 'low': 'min',
    'close': 'last', 'volume': 'sum'
}).dropna()
```

**数据读取：** 通过 `DataPipeline.get_bars(trading_date=scan_date)` 读取，继承前瞻偏差防护。不绕过 DataPipeline 直接查 DuckDB。

**筛选条件（全部用 Python 计算，无 AI）：**
1. 成交额过滤：过去 20 日均成交额 > 1000 万（排除流动性差的股票）
2. 周线 EMA(26) 方向：上升或下降（排除走平）
3. 周线 MACD 柱：判断季节（spring/summer/autumn/winter）
4. 日线随机指标：做多场景 < 30，做空场景 > 70
5. 日线 2 日强力指数：做多场景为负，做空场景为正

**评分公式（`score` 字段）：**
- 基础分：通过条件 2-5 各 +2 分（满分 8 分）
- 加分：条件 3 为 spring/autumn（最佳季节）额外 +1 分
- 加分：条件 4 和 5 同时满足（双重确认）额外 +1 分
- 最高 10 分，按 score 降序排列

**指标计算：** 使用 `pandas_ta`（纯 Python，无 C 编译依赖）。

**最小数据要求：** 周线数据 < 52 周的股票直接跳过，计入 `filtered_out.insufficient_data`。52 周保证 EMA(26) 有足够的初始化缓冲。

#### scan-canslim：基本面指标过滤

**数据需求：**
- 财务数据：需要预先持久化到本地（见"基本面数据存储"节）
- 技术数据：本地 DuckDB 日线数据（用于计算相对强度）

**基本面数据存储（新增前置工作）：**

`fundamental` 命令目前只打印，不持久化。需新增 `fundamental-store` 命令，本质是对每个符号调用已有的 `fundamental_source.get_financial_summary(symbol_id, years=5)` 并把结果写入磁盘，不需要重新实现数据获取逻辑：

```
data/fundamental/
  {exchange}_{ticker}.json    # 如 SSE_600000.json、SZSE_000001.json
```

文件名规则：`{exchange}_{ticker}.json`（用下划线，避免冒号在 Windows 上非法）。
`load_fundamental("SSE:600000")` → 读取 `data/fundamental/SSE_600000.json`。

scan-canslim 从这个目录批量读取，不实时调 API。

`fundamental-store` 的错误处理：单只股票失败时跳过（记录日志），继续其他。完成后输出汇总：成功 N 只，失败 M 只。

`fundamental-store` 的幂等性：
- 默认行为：覆盖已有文件（季报后刷新场景）
- `--skip-existing` 标志：跳过已有文件（增量更新，只处理新股票）

**EPS 增速定义：** 同比口径（YoY）。
```
当季 EPS 增速 = (本季 EPS - 去年同期 EPS) / |去年同期 EPS|
```
**边界处理：** 去年同期 EPS ≤ 0 时，无法计算有意义的增速，将该股票计入 `filtered_out.insufficient_data`，跳过。不设为 +inf（避免过滤逻辑失效）。

**筛选条件：**
1. 当季 EPS 增速（同比）≥ 18%（C 维度硬性过滤线）
2. 年度 EPS 连续增长（过去 3 年无亏损年，A 维度）
3. ROE ≥ 17%
4. 相对强度排名：过去 52 周涨幅相对全市场排名前 20%（L 维度代理指标，从日线数据计算）
5. 大盘状态：如果沪深 300 周线 EMA 下降，输出警告（不阻止扫描）

**数据缺失处理：** 无 `data/fundamental/{ticker}.json` 的股票计入 `filtered_out.insufficient_data`，跳过，不报错。

**注意：** 基本面数据有延迟（季报），scan-canslim 结果反映最新财报，不是实时数据。

#### scan-value：估值指标过滤

**数据需求：**
- 当前 PE、PB、市值：`AkshareFactorSource.get_stock_basic_info()`（调用 `ak.stock_zh_a_spot_em()`，已有）
- ROE：`data/fundamental/{ticker}.json`（与 scan-canslim 共用同一存储）
- 历史 PE 分位：从本地 DuckDB 日线数据 + 当前 EPS 近似计算（见下）

**PE 历史分位计算方案（MVP 简化版）：**

精确 PE 历史分位需要历史 EPS 时间序列，当前系统没有。MVP 阶段用代理指标：

```
PE 历史分位 ≈ 当前 PE 在过去 N 年收盘价序列中的分位
（用价格分位代替 PE 分位，假设 EPS 变化相对价格变化较小）
```

这是近似值，对于 EPS 波动大的公司（周期股）不准确。在输出 JSON 中标注 `"pe_percentile_method": "price_proxy"`。

如需精确 PE 历史分位，需要先完成 `fundamental-store` 并积累历史季度 EPS 数据（不在本期范围内）。

**筛选条件：**
1. 当前 PE 相对该股过去 3 年价格序列的分位 < 30%（价格代理法，见上）；数据不足 1 年的跳过，1-3 年的标注"数据有限"
2. PB < 3（排除高溢价，来自 `stock_zh_a_spot_em()`）
3. ROE ≥ 15%（来自 `data/fundamental/`，无数据则跳过）
4. 市值 > 50 亿（来自 `stock_zh_a_spot_em()`）

**注意：** ST / 退市股在 `get_scan_symbols()`（common.py）层面统一过滤，各 scanner 不重复处理。

**PE/PB 数据说明：** `stock_zh_a_spot_em()` 返回实时快照，不是 scan_date 的历史数据。这是已知的近似，在输出 JSON 里标注 `"pe_pb_source": "realtime_snapshot"`。PE/PB 变化远慢于价格，这个近似对于日线级别的扫描是可接受的。

### 代码结构

```
src/trading_os/
├── cli.py                          # 新增四个 CLI 命令入口（scan-elder/canslim/value + fundamental-store）
├── scan/                           # 新建模块
│   ├── __init__.py
│   ├── common.py                   # 共享工具（见下）
│   ├── elder_scanner.py            # Elder 体系筛选逻辑
│   ├── canslim_scanner.py          # CANSLIM 体系筛选逻辑
│   └── value_scanner.py            # Value Investing 体系筛选逻辑
└── data/
    ├── sources/
    │   └── akshare_factors.py      # 已有：get_a_stock_list() + get_stock_basic_info()
    │                               # common.py 直接调用，不新建 market_snapshot.py
    └── fundamental/                # 新建：基本面数据存储目录
        └── {ticker}.json           # 每只股票一个文件（由 fundamental-store 命令写入）
```

**`common.py` 的职责（共享工具层）：**
- `get_scan_symbols(pipeline, exchange)` → 两步过滤：
  1. 调用 `AkshareFactorSource.get_a_stock_list()` 获取全市场列表，过滤 ST/退市
  2. 将 AKShare 格式（`exchange="SSE"`, `symbol="600000"`）转换为规范格式（`"SSE:600000"`）
     转换函数：`to_canonical(exchange: str, ticker: str) -> str`，如 `to_canonical("SSE", "600000") → "SSE:600000"`
  3. 与 `DataPipeline.available_symbols(exchange=exchange)` 取交集，只扫描本地有数据的
  4. 不在本地的记入 `filtered_out.no_data`（不报错，正常情况）
- `filter_by_turnover(symbols, min_amount)` → 成交额（turnover，单位 CNY）过滤，**不叫 min_volume**
- `load_bars_batch(pipeline, symbols, scan_date, lookback_days)` → 调用 `DataPipeline.get_bars(symbols=batch, trading_date=scan_date)`，一次调用返回整批数据，继承前瞻偏差防护
- `write_scan_output(results, path)` → 统一 JSON 输出格式
- `load_fundamental(ticker)` → 读取 `data/fundamental/{ticker}.json`，文件不存在返回 None

**内存处理：** 各 scanner 按批次处理（每批 500 只），调用 `load_bars_batch(batch_of_500)` 一次返回整批 DataFrame，不逐只调用。

**AKShare 网络错误处理：** `get_scan_symbols()` 和 `scan-value` 的 `stock_zh_a_spot_em()` 调用都应捕获网络异常，以非零退出码退出，输出清晰错误信息（如"AKShare 不可用，请检查网络连接"）。不静默崩溃，不重试。

**各 scanner 的职责（业务层）：**
- 只包含该体系特定的筛选逻辑
- 不包含数据获取代码（调用 common.py）
- 不包含 AI 判断（那是 skill 层的事）

### skill 层的适配

三个 skill 文件需要更新，各自增加"读取扫描结果"触发模式：

| Skill 文件 | 新增触发词 |
|-----------|-----------|
| `elder/signal-scanner/SKILL.md` | "读取 Elder 扫描结果"、"分析昨天的 Elder 扫描" |
| `canslim/canslim-system/SKILL.md` | "读取 CANSLIM 扫描结果"、"分析 CANSLIM 候选清单" |
| `value-investing/value-system/SKILL.md` | "读取价值投资扫描结果"、"分析估值筛选结果" |

每个 skill 新增模式 B 的处理逻辑：
```
模式 A（原有）：用户提供候选池 → skill 做 AI 分析
模式 B（新增）：读取 artifacts/scan/{system}-{date}.json
              → skill 对候选清单逐一做深度 AI 分析
              → 输出格式与模式 A 相同（Markdown 分析报告）
```

skill 读 JSON 文件，输出 Markdown 分析报告给用户。JSON 是输入协议，不是最终输出格式。

**路径约定：** `--output` 默认值为 `{repo_root}/artifacts/scan/{system}-{date}.json`（相对于 repo root，不是当前工作目录）。CLI 命令通过 `trading_os.paths` 模块获取 repo root。Skill 层读取时使用同样的路径约定。

## 数据依赖和前置条件

| 体系 | 数据需求 | 前置条件 |
|------|---------|---------|
| Elder | 本地 DuckDB 日线/周线 | 需要先 `fetch-ak` 补充历史数据（建议 2 年） |
| CANSLIM | BaoStock 基本面 + 本地 K 线 | 需要先 `fundamental` 拉取财务数据 |
| Value Investing | PE/PB 历史数据 | 需要先 `fetch-ak` 补充至少 3 年历史数据 |

首次使用建议：
```bash
# 1. 初始化数据湖
python -m trading_os lake-init

# 2. 逐只拉取历史数据（现有命令，需要循环调用）
# fetch-ak 目前不支持 --all-a-share，需要脚本循环：
for ticker in $(python -m trading_os list-symbols --exchange SSE); do
  python -m trading_os fetch-ak --exchange SSE --ticker $ticker --start 2022-01-01
done
# 注意：这是当前的临时方案，全市场批量拉取命令（fetch-ak-bulk）是后续优化项

# 3. 持久化基本面数据（新命令，本期新增）
python -m trading_os fundamental-store --all  # 写入 data/fundamental/

# 4. 运行扫描
python -m trading_os scan-elder --top 30
python -m trading_os scan-canslim --top 30    # 需要先完成步骤 3
python -m trading_os scan-value --top 30      # 需要先完成步骤 2 和 3
```

## 成功标准

- `scan-elder --top 30` 在 5 分钟内完成全 A 股扫描，输出 JSON 文件（假设本地有 2 年日线数据）
- `scan-canslim --top 30` 正确过滤出当季 EPS 同比增速 ≥ 18% 的股票（需要先运行 `fundamental-store`）
- `scan-value --top 30` 正确计算价格代理 PE 分位并过滤（MVP 精度，非精确 PE 历史分位）
- skill 层读取 JSON 后能输出 Markdown 分析报告，无需手动格式转换
- 三个 scan 命令的代码互不依赖（可以独立修改任意一个）
- `filtered_out` 字段包含 `no_data`、`low_turnover`、`no_signal`、`insufficient_data` 四类

## Open Questions

1. **全市场历史数据批量获取**：`fetch-ak` 目前逐只调用，5000 只串行耗时较长。建议先测量实际耗时，再决定是否新增 `fetch-ak-bulk` 并行命令。这是后续优化项，不阻塞 scan-elder 的 MVP。

2. **`fundamental-store` 的更新策略**：基本面数据多久需要刷新一次？建议：季报发布后（每年 4 月、8 月、10 月、次年 4 月）手动触发，不自动定时更新。

3. **scan-value 的精确 PE 历史分位**：MVP 用价格代理法，精度有限。后续当 `fundamental-store` 积累了足够的历史季度 EPS 数据后，可以升级为精确计算。这是明确的技术债，不是遗漏。
