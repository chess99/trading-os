## Problem Statement

现有 skills 架构把 Elder（技术交易）、CANSLIM（成长股）、巴菲特/格雷厄姆（价值投资）三套体系的 skills 混在同一目录层级，由 `trading-system` 统一调度。

这个设计有三个问题：
1. **操作逻辑冲突**：Elder 用价格止损（2%/6%），巴菲特用逻辑止损，CANSLIM 两者结合。三套止损逻辑混在一起，`position-monitor` 无法清晰处理。
2. **心理模型冲突**：技术分析要求关注价格行为，价值投资要求忽略短期价格波动。两种思维模式同时激活会互相干扰。
3. **迭代困难**：三套体系耦合，改一处可能影响另外两处，无法独立打磨每套体系到极致。

---

## 核心洞察

不同流派不是"工具箱里的工具"，而是完整的世界观，包含互相矛盾的心理模型。

解决方案：**按体系完全拆分，账户层面隔离**。
- 一个交易账号完全走一套体系
- 每套体系有自己的入口 skill、分析工具、仓位逻辑、交易指令
- 真正需要共享的只有数据获取层（AKShare/BaoStock）

---

## 约束

- 现有 trading-os 代码层（backtest/paper/risk）不动，只重组 `.claude/skills/` 目录
- 三套体系各自独立，不强制互相调用
- 保留 `trading-system` 作为顶层入口，但它的职责变成"路由到哪套体系"，而不是"编排多套体系"

---

## 前提

1. 不同流派是完整世界观，同时运行多套会互相干扰——已确认
2. trading-os 代码系统是执行层，skills 是决策层——已确认
3. 耦合不利于迭代，分开才能把独立体系打造到极致——已确认（用户明确表达）
4. 账户层面隔离：一个账号完全走一套体系——已确认
5. 先建三个体系，巴菲特/格雷厄姆体系可以先建骨架，后续深入——已确认

---

## 方案 A：双轨制（最小改动）

只改 `trading-system` 的调度逻辑，声明两个独立子系统（技术账户 30% + 成长股账户 70%）。

- 努力：人工 ~1天，Claude Code ~15分钟
- 完整度：7/10
- 缺点：两套止损逻辑仍然共享 `position-monitor`，混乱根源未解决

## 方案 B：完全拆分（选定方案）

三套完全独立的 skill 子目录，各有入口、分析、仓位、执行工具。

- 努力：人工 ~1周，Claude Code ~2小时
- 完整度：9/10
- 优点：真正隔离，可以独立迭代，止损逻辑清晰归属

## 方案 C：三层架构

研究层 + 交易层 + 编排层，两种止损逻辑有明确归属，但三套体系仍然耦合。

- 努力：人工 ~2天，Claude Code ~30分钟
- 完整度：9/10
- 缺点：仍然耦合，不利于独立迭代

---

## 推荐方案：方案 B 完全拆分

用户选择理由：耦合不利于迭代，分开才能把独立体系打造到极致，账户层面也会隔离。

## 目标目录结构

**当前实际存在的 skills**（`git mv` 的来源）：
```
.claude/skills/
  backtest-review/  canslim-screen/  elder-screen/  fundamental-research/
  position-monitor/ position-sizer/  signal-scanner/ trade-executor/
  trading-journal/  trading-system/
    trading-system/references/config.md
    trading-system/references/interfaces.md
```

注意：`valuation/` 目前**不在** `.claude/skills/` 下，是 Python CLI 命令，不是 skill。Phase 4 需要**新建** `value-investing/valuation/SKILL.md` 来封装它，不是移动。

**目标结构**：
```
.claude/skills/

│   ├── SKILL.md             ← 职责：识别意图，输出"请使用 elder-system / canslim-system / value-system"指令
│   └── references/          ← 保留：interfaces.md 升级为三套体系的共享接口契约
│       ├── config.md
│       └── interfaces.md    ← 更新：记录三套体系各自的输入/输出格式
│
├── elder/                   ← Elder 技术交易体系（完全独立）
│   ├── elder-system/        ← 体系入口（新建）—— 触发词见下方路由表
│   ├── elder-screen/        ← git mv 从根目录
│   ├── signal-scanner/      ← git mv 从根目录
│   ├── position-sizer/      ← git mv 从根目录（保留 2%/6% 逻辑，确认无 CANSLIM 内容）
│   ├── trade-executor/      ← git mv 从根目录
│   ├── position-monitor/    ← git mv 从根目录（修改：只保留价格止损逻辑）

│   └── backtest-review/     ← git mv 从根目录

├── canslim/                 ← CANSLIM 成长股体系（完全独立）
│   ├── canslim-system/      ← 体系入口（新建）—— 触发词见下方路由表
│   ├── canslim-screen/      ← git mv 从根目录
│   ├── fundamental-research/← git mv 从根目录（初期与 value-investing 共享同一份内容）
│   ├── elder-confirm/       ← 新建：从 elder-screen 复制后裁剪，只保留第二/三滤网（确认用途）

│   ├── trade-executor/      ← 新建
│   ├── position-monitor/    ← 新建：双模式止损（初期价格止损，盈利后逻辑止损）
│   └── trading-journal/     ← 新建

└── value-investing/         ← 巴菲特/格雷厄姆价值投资体系（骨架先建）

    ├── fundamental-research/← 新建：与 canslim/fundamental-research 初期内容相同，后续按需分叉
```

**`fundamental-research` 的处理决策**：
- `canslim/fundamental-research/` 和 `value-investing/fundamental-research/` 是两份独立文件（不共享物理文件）
- 初期内容相同（从根目录 `fundamental-research/` 复制两份），然后 `git rm` 根目录版本
- 后续随各体系需求独立演化，不需要手动同步
- 这是有意的内容重复，为了换取独立迭代能力

**`elder-confirm` 的处理决策**：
- 从 `elder/elder-screen/SKILL.md` 复制后裁剪：只保留第二滤网（日线动量确认）和第三滤网（精确入场）
- 删除第一滤网（周线趋势判断），因为 CANSLIM 流程里基本面研究已经起到趋势过滤作用
- `elder/elder-screen` 和 `canslim/elder-confirm` 是独立副本，elder-screen 变更时手动评估是否需要同步 elder-confirm

**`trading-journal` 的处理决策**：
- 三套体系各自独立的 journal（内容格式和复盘逻辑不同）
- Elder：价格止损触发、技术指标复盘、系统规则遵守情况
- CANSLIM：基本面变化追踪、持仓阶段记录、逻辑止损条件更新
- Value Investing：买入理由记录、护城河评估更新、估值变化追踪

**`trading-system/references/` 的处理**：
- `config.md`：更新为三套体系各自的配置节（账户参数、候选股池），不再共享一个全局配置
- `interfaces.md`：更新为三套体系各自的输入/输出格式契约，以及 `trading-system` 路由输出的格式

**`CLAUDE.md` 更新**：
- Phase 1 完成后必须更新项目 `CLAUDE.md` 中的 skills 表，把旧的平铺路径改为新的子目录路径，否则 Claude 加载 CLAUDE.md 时会看到不存在的 skill 路径

**`trading-system/SKILL.md` 中的"顶级交易员共同原则"内容**：
- 这部分内容（来自施瓦格《市场奇才》）与路由逻辑无关
- 迁移到 `trading-system/references/principles.md`，三套体系的入口 skill 可以按需引用

**`trading-system` 的路由机制（已决定）**：
Claude Code 的 skill 触发是基于用户意图匹配 SKILL.md 的 description 字段，不是程序化调用。`trading-system` 无法直接"调用" `elder-system`。

**选定方案：输出建议，用户触发**
- `trading-system` 识别用户意图，输出路由建议：
  - "您的请求适合 **Elder 技术交易体系**。请直接说：'用 Elder 分析 600000' 进入该体系。"
  - 或者更简洁：列出三套体系的简介和触发词，让用户选择
- 用户看到建议后，用对应体系的触发词重新发起请求，触发 `elder-system` / `canslim-system` / `value-system`
- 这保持了三套体系的真正独立性，`trading-system` 只是一个"导航页"，不嵌入任何体系的逻辑

**注意**：`trading-system` 的 description 字段应该匹配"意图不明确"的宽泛触发词（如"分析一下这只股票"、"帮我看看600000"），而三套体系的入口 skill 匹配各自明确的触发词。这样用户有明确意图时直接进入体系，没有明确意图时先经过路由。

**三套体系的触发词路由表**：

| 体系 | 入口 skill | 触发词示例 |
|------|-----------|-----------|
| Elder 技术交易 | `elder/elder-system` | "用 Elder 分析 600000"、"三重滤网分析"、"技术面分析"、"这个位置能进吗" |
| CANSLIM 成长股 | `canslim/canslim-system` | "CANSLIM 评分"、"成长股筛选"、"基本面+技术面综合分析"、"这股票值得长期关注吗" |
| Value Investing | `value-investing/value-system` | "估值分析"、"内在价值"、"巴菲特框架"、"这公司护城河怎么样" |
| 路由入口 | `trading-system` | "分析 600000"（意图不明确时，先路由）、"帮我看看这只股票" |

## 各体系的止损逻辑

### Elder 体系（价格止损）
- 每笔交易入场前设置止损位（通常在近期低点或 ATR 倍数以下）
- 单笔风险不超过账户 2%
- 月度总风险不超过账户 6%（触发后停止新开仓）
- `position-monitor` 只检查价格是否触及止损位

### CANSLIM 体系（双模式止损）
- **入场后初期**：价格止损（通常 7-8% 跌幅，欧奈尔规则）
- **持仓盈利后**：逻辑止损为主（基本面恶化、行业地位下降、机构撤离）
- `position-monitor` 需要知道持仓阶段（初期/盈利期）

### Value Investing 体系（纯逻辑止损）
- 没有价格止损
- 卖出条件：买入理由不再成立（竞争优势消失、管理层变质、估值严重高估）
- `position-monitor` 定期检查基本面，不检查价格

## 实现步骤

### Phase 0（前置验证）：确认 Claude Code 的 skill 发现机制

**目标**：在大规模移动文件之前，先验证嵌套目录是否被 Claude Code 正确识别。

1. 创建测试目录：`mkdir -p .claude/skills/elder/elder-system`
2. 新建 `elder/elder-system/SKILL.md`，只包含 description 字段和一句占位内容
3. 测试：在 Claude Code 中说"测试 elder system"，看是否能触发这个 skill
4. **如果嵌套目录不被识别**：改用命名前缀方案（不移动目录，改用 `elder.elder-screen` 等命名），整个架构设计不变，只是物理组织方式不同
5. 验证通过后删除测试文件，进入 Phase 1

### Phase 1：建立目录骨架 + 重写路由入口

**目标**：移动文件，重写 trading-system，Phase 1 结束后 trading-system 能正确路由。

1. 创建子目录：`mkdir -p .claude/skills/{elder,canslim,value-investing}`
2. `git mv` 现有 skills 到对应目录：
   - Elder：elder-screen, signal-scanner, position-sizer, trade-executor, position-monitor, trading-journal, backtest-review
   - CANSLIM：canslim-screen
   - 根目录 fundamental-research：**复制**两份到 `canslim/fundamental-research/` 和 `value-investing/fundamental-research/`，然后 `git rm` 根目录版本
3. **完全重写** `trading-system/SKILL.md`（不是小修改）：
   - 删除现有的六套编排流程（A-F）
   - 迁移"顶级交易员共同原则"内容到新建的 `references/principles.md`
   - 新内容：识别意图 → 输出路由建议（列出三套体系的简介和触发词）
   - 更新 `references/config.md`：拆分为三套体系各自的配置节
   - 更新 `references/interfaces.md`：记录三套体系各自的输入/输出格式
4. **更新 `CLAUDE.md`**：把 skills 表中的旧路径更新为新的子目录路径
5. **Phase 1 验证标准**：说"分析 600000"，trading-system 能输出明确的路由建议（列出三套体系和对应触发词），而不是直接执行分析。此时三套体系的入口 skill 还不存在，这是预期的。

### Phase 2：完善 Elder 体系（最成熟，先完善）

1. 新建 `elder/elder-system/SKILL.md`：
   - 触发词：见路由表
   - 内容：Elder 三重滤网完整流程（原 trading-system 的技术分析流程迁移过来）
2. 修改 `elder/position-monitor/SKILL.md`：删除所有逻辑止损相关内容，只保留价格止损
3. 确认 `elder/position-sizer/SKILL.md`：只用 2%/6% 原则（ATR 止损），无 CANSLIM 内容
4. **Phase 2 验证标准**：说"用 Elder 分析 600000"，完整走完三重滤网流程，只调用 `elder/` 下的 skills

### Phase 3：完善 CANSLIM 体系

1. 新建 `canslim/canslim-system/SKILL.md`：
   - 触发词：见路由表
   - 内容：CANSLIM 完整流程（基本面评分 → 技术面确认 → 仓位计算 → 指令生成）
2. 新建 `canslim/elder-confirm/SKILL.md`：
   - 从 `elder/elder-screen/SKILL.md` 复制，裁剪掉第一滤网（周线趋势），只保留第二、三滤网
   - 在文件头注明："此 skill 是 elder-screen 的裁剪版，专用于 CANSLIM 流程的技术面确认步骤"
3. 新建 `canslim/position-sizer/SKILL.md`：
   - 欧奈尔规则：初始止损 7-8%（不同于 Elder 的 ATR 止损），据此计算仓位大小
4. 新建 `canslim/position-monitor/SKILL.md`：
   - 双模式：持仓阶段字段（`stage: early | profitable`），early 用价格止损，profitable 用逻辑止损
5. 新建 `canslim/trade-executor/SKILL.md`（成长股交易指令）
6. 新建 `canslim/trading-journal/SKILL.md`（成长股交易记录）
7. **Phase 3 验证标准**：说"CANSLIM 分析 600000"，只调用 `canslim/` 下的 skills，不触发 `elder/` 下的任何 skill

### Phase 4：建立 Value Investing 体系骨架

1. 新建 `value-investing/value-system/SKILL.md`（体系入口，包含使用说明和触发词）
2. 新建 `value-investing/valuation/SKILL.md`：封装 `python -m trading_os valuation` 命令
3. 新建 `value-investing/position-monitor/SKILL.md`：纯逻辑止损，不检查价格
4. **Phase 4 验证标准**：说"估值分析茅台"，调用 value-investing 体系，能运行 valuation 命令
