## Problem Statement

构建一个可靠的**回测-模拟-实盘闭环**量化交易系统，专注 A 股市场。

核心要求：
1. 策略一次写，三个环境（回测/模拟/实盘）跑——不改代码，只换配置
2. AI/Agent 原生集成——Claude Agent 不是外挂，是策略生成和分析的一等公民
3. 支持全自动下单（类似 Claude Code bypassPermissions 模式）
4. 多因子选股 + 任意策略类型（不限于定投）
5. 数据质量是生命线——严格前瞻偏差防护，宁报错不用脏数据

---

## What Makes This Different

其他 7 个调研项目（TradingAgents、Qbot、go-stock 等）都把"AI 分析"和"回测引擎"当两套独立系统建。

**EUREKA 洞察**：策略本身就是 AI 的输出物。AI 说"买"，但这个信号没经过回测验证，AI 分析等于零。正确架构是：**AI 生成策略参数 → 策略在回测引擎里验证 → 通过验证才能在模拟/实盘里跑**。这是一个闭环，不是两个独立系统。

---

## Constraints

- 专注 A 股（沪深），架构支持未来扩展
- 全部模拟阶段，暂无真实资金（但设计必须支持实盘）
- 数据源：AKShare（免费）+ Tushare Pro（付费备选）
- Claude Code 军团支持，代码规模不是瓶颈
- `vendor/` 中有 7 个开源参考库，可直接借鉴或复制

## Recommended Approach: 事件驱动 + Agent 原生架构

### 系统分层

```
┌─────────────────────────────────────────────────────────┐
│                    用户层 / CLI                          │
│  trading-os backtest / paper / live / agent             │
├─────────────────────────────────────────────────────────┤
│                   Agent 决策层（Phase 3）                │
│  ClaudeAgent → 分析市场 → 生成/调整策略参数              │
```

### 核心设计决策

#### 1. 统一策略接口（最重要）

```python
class Strategy(ABC):
    """三环境统一策略基类。同一份代码在回测/模拟/实盘里跑。"""

    @abstractmethod
    def generate_signals(self, bars: DataFrame, curr_date: date) -> dict[str, Signal]:
        """
        输入：截止 curr_date 的历史 K 线（严格不含未来数据）
        输出：{symbol: Signal(action=BUY/SELL/HOLD, size=0.1, reason="...")}
        """
        ...

    def on_fill(self, fill: Fill) -> None:
        """成交回调，用于更新内部状态"""
        pass

    def on_start(self, context: StrategyContext) -> None:
        """策略启动时调用"""
        pass
```

**关键约束**：`generate_signals` 接收的 `bars` 由引擎保证只包含 `curr_date` 之前的数据。策略不需要自己做日期过滤。

#### 2. 事件驱动引擎

```python
# 事件流（三环境完全一致）
BarEvent(symbol, bar)
  → Strategy.generate_signals()
  → SignalEvent(symbol, signal)
  → RiskManager.check(signal, portfolio)  # 硬性守门人
  → OrderEvent(symbol, order)
  → Broker.execute(order)  # 三环境唯一不同的地方
  → FillEvent(fill)
  → Portfolio.update(fill)
  → EventLog.record(fill)
```

三个环境的区别**只在 Broker**：
- `BacktestBroker`：用历史 open 价成交，模拟滑点/手续费
- `PaperBroker`：用实时行情，虚拟账户
- `LiveBroker`：接真实券商 API（XTP/掘金/东财等）

#### 3. AI Agent 集成（原生，不是外挂）

```python
class AgentStrategy(Strategy):
    """Claude Agent 驱动的策略。"""

    def generate_signals(self, bars: DataFrame, curr_date: date) -> dict[str, Signal]:
        # 1. 准备市场上下文（技术指标 + 基本面摘要）
        context = self._build_context(bars, curr_date)

        # 2. 调用 Claude Agent 分析
        analysis = self.agent.analyze(context)

        # 3. 解析 Agent 输出（Pydantic 严格验证）
        return self._parse_signals(analysis)

    # 确认模式（默认）
    confirm_mode: Literal["auto", "confirm"] = "confirm"
    # confirm=True → 显示 Agent 分析，等待用户确认后执行
    # confirm=False → 全自动执行（bypassConfirm）
```

Agent 层设计：
- **结构化输出**：Pydantic 严格验证，不依赖 LLM 自由格式文本
- **解析失败策略**：Pydantic 验证失败时，对所有标的发出 HOLD 信号并记录 WARNING；连续 3 次失败则停止策略并报错
- **多 Agent 辩论 + BM25 记忆**：Phase 4+ 可选扩展，Phase 3 只做单 Agent

#### 4. 风控层（量化守门人，不是 LLM）

```python
class RiskManager:
    """硬性风控，不可被 AI 绕过。"""

    def check(self, signal: Signal, portfolio: Portfolio) -> OrderDecision:
        # 以下任何一条触发 → 拒绝信号
        if self._exceeds_position_limit(signal, portfolio):  # 单股持仓上限 10%
            return OrderDecision.REJECT("超过单股持仓上限")
        if self._exceeds_sector_concentration(signal, portfolio):  # 行业集中度 30%
            return OrderDecision.REJECT("行业集中度超限")
        if self._var_breach(portfolio):  # 日 VaR 超限
            return OrderDecision.REJECT("VaR 超限")
        if self._circuit_breaker(portfolio):  # 日亏损熔断
            return OrderDecision.REJECT("日亏损熔断触发")
        return OrderDecision.APPROVE(signal)
```

**关键设计**：AI 建议的信号必须通过量化风控才能执行。AI 说"买"，但 VaR 超限，就不买。

#### 6. 多因子选股

```python
class FactorLibrary:
    """内置因子库，策略可以组合使用。"""

    # 技术因子
    momentum_1m: Factor  # 1 个月动量
    rsi_14: Factor       # RSI(14)
    macd_signal: Factor  # MACD 信号

    # 基本面因子
    pe_ratio: Factor     # 市盈率
    roe: Factor          # 净资产收益率
    revenue_growth: Factor  # 营收增速

    # 资金流因子（A 股特有）
    north_flow: Factor   # 北向资金
    main_force_flow: Factor  # 主力资金流

class MultiFactorStrategy(Strategy):
    def __init__(self, factors: list[tuple[Factor, float]]):
        """factors: [(因子, 权重), ...]"""
        ...

    def generate_signals(self, bars, curr_date) -> dict[str, Signal]:
        scores = self._compute_factor_scores(bars, curr_date)
        top_n = scores.nlargest(self.n_stocks)
        return {sym: Signal.BUY for sym in top_n.index}
```

### 技术选型

| 层次 | 选择 | 理由 |
|------|------|------|
| 数据存储 | DuckDB + Parquet | 列式存储，查询快，现有代码已用 |
| 账户/订单存储 | SQLite | 轻量，无服务端依赖 |
| 数据源（主） | AKShare | 免费，A 股覆盖全 |
| 数据源（备） | Tushare Pro | 稳定，需 Token |
| AI 层 | Claude API（原生） | 不走 LangChain，直接调用 |
| 策略记忆 | BM25（rank-bm25） | Phase 4+ 可选扩展 |
| 输出验证 | Pydantic v2 | 严格验证 AI 输出 |
| CLI | Typer + Rich | 现代化 CLI，进度展示 |
| 测试 | pytest | 标准 |

**不用 LangChain/LangGraph**：直接调用 Claude API，减少抽象层。Agent 编排自己实现，更可控。

## 关键设计决策（补充）

### Signal 数据结构
```python
@dataclass
class Signal:
    symbol: str
    action: Literal["BUY", "SELL", "HOLD"]
    size: float          # 目标持仓占 portfolio NAV 的比例，0.0~1.0
    reason: str          # 人类可读的理由
    confidence: float    # 0.0~1.0，AI 策略专用
    valid_until: date    # 信号有效期（过期自动失效）
```

`size` 语义：占组合净值的目标比例。`size=0.1` 表示该标的目标仓位为组合净值的 10%。Portfolio 负责计算实际需要买卖多少股。

### Portfolio 状态
```python
@dataclass
class Portfolio:
    cash: float
    positions: dict[str, Position]  # symbol → Position
    nav: float                       # 净值（cash + 所有持仓市值）

@dataclass
class Position:
    symbol: str
    shares: float
    avg_cost: float      # 成本价（已考虑历史分红/拆股）
    market_value: float  # 当前市值
```

A 股除权除息处理：BacktestBroker 使用前复权数据，持仓成本自动调整。LiveBroker 从券商 API 获取真实成本。

### curr_date 语义
`curr_date` = 当前交易日 T。`get_bars(as_of=curr_date)` 返回 T-1 收盘及之前的数据（A 股日线数据在 T 日 15:30 后才完整，因此 T 日只能用 T-1 及之前的数据）。这个约束由 `DataPipeline` 强制执行，策略无需关心。

### Paper Trading 数据源
**明确约定**：模拟交易是**日线级别**，不支持日内交易。
- 使用 AKShare 日线数据（T+1 日早上可获取 T 日收盘数据）
- 每日收盘后运行一次策略，次日开盘执行
- 这与回测的执行模型完全一致（信号在 T 收盘计算，T+1 开盘执行）

### Agent API 调用经济性
`AgentStrategy` 不在每根 K 线上调用 Claude。调用频率：
- **默认**：每日收盘后调用一次（分析当日行情，生成次日信号）
- **批量**：一次 API 调用分析整个自选股池（而非每个标的单独调用）
- **缓存**：相同 `curr_date` 的分析结果缓存到磁盘，重跑不重复调用

### 数据源优先级（统一）
A 股数据源优先级（从高到低）：
1. **Tushare Pro**（付费，最稳定，日线/分钟线/基本面）
2. **AKShare**（免费，东方财富，日线/基本面）
3. **BaoStock**（免费，历史数据）
4. **本地缓存**（DuckDB，离线可用）

触发 Fallback 的条件：HTTP 错误、超时（5s）、数据为空、数据质量校验失败（如收盘价为 0）。

### A 股市场规则（BacktestBroker 必须实现）

这些规则不是可选的。缺少任何一条，回测结果都是不可信的：

| 规则 | 说明 | 实现方式 |
|------|------|---------|
| **T+1 结算** | 当日买入的股票，次日才能卖出 | Portfolio 记录每笔持仓的 `buy_date`，卖出时检查 `curr_date > buy_date` |
| **涨跌停限制** | 普通股每日涨跌幅 ±10%（ST 股 ±5%） | 订单价格超过涨跌停价则拒绝，并记录 ORDER_REJECTED |
| **最小交易单位** | 买入必须是 100 股的整数倍（1 手） | 下单时向下取整到最近的 100 股整数倍 |
| **科创板/创业板** | 注册制股票涨跌停 ±20%，新股前 5 日无涨跌停 | 按股票类型动态确定涨跌停幅度 |
| **停牌处理** | 停牌期间无法交易 | 数据层标记停牌状态，BacktestBroker 拒绝停牌股的订单 |


`--bypass-confirm` CLI flag 对应 `confirm_mode: auto`（全自动，无人值守）。

### 审计日志实现
`EventLog` 使用 SQLite append-only 表（代码层面不提供 UPDATE/DELETE 接口）。每条记录包含：
- `id`（自增）
- `ts`（时间戳）
- `event_type`（BAR/SIGNAL/ORDER/FILL/RISK_REJECT/SIGNAL_EXPIRED）
- `payload`（JSON）
- `created_at`（写入时间，与 ts 区分）

不做 SHA256 链式结构（YAGNI，单人项目无审计合规需求）。

### 范围边界（明确不做的事）
- **不做**多 Agent 辩论框架（Phase 3 只做单 Agent，辩论框架是未来可选项）
- **不做** Web UI（CLI 优先，Web UI 是独立项目）
- **不做** BM25 记忆（Phase 3 先验证单 Agent 效果，记忆是优化项）
- **不做**日内交易（日线级别，模拟和回测模型一致）
